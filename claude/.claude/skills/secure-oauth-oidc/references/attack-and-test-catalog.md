# OAuth/OIDC Attack and Test Catalog

Use this catalog to move from a control name to a concrete attacker story and negative test. It tracks [RFC 9700 §3–§4](https://www.rfc-editor.org/rfc/rfc9700.html#section-3); follow the linked RFC section for full preconditions and variants.

## Attacker model

| Attacker | Capability to model | Important limit / implication |
|---|---|---|
| **A1 — web attacker** | Operates arbitrary endpoints, sites, clients, authorization servers, resource servers, browsers, and accounts; can lure a victim to chosen URLs and replay values already learned. | Cannot read or alter traffic not addressed to attacker-controlled endpoints when cryptography is correctly applied. Do not assume an attacker lacks a valid client or account. |
| **A2 — network attacker** | Eavesdrops, modifies, spoofs, reorders, or blocks traffic not correctly protected by cryptography. | Proper TLS blocks reading/modification, but not blocking. Model proxy/internal-hop gaps separately. |
| **A3 — authorization-response reader** | Learns codes, state, access tokens, or other contents of an authorization response without necessarily modifying it. | Models browser history, Referer leakage, redirect mistakes, mobile URI collisions, logs, and compromised endpoints. A code is not harmless because it is short-lived. |
| **A4 — authorization-request reader** | Learns request contents, including a `plain` PKCE challenge or `nonce`. | Motivates `S256`, transaction-specific values, and protection against request leakage/tampering. |
| **A5 — access-token holder** | Acquires a valid token through leakage, phishing, misconfiguration, or compromise of a resource server. | Assume replay is attempted from a different client or resource. Audience and sender constraint limit blast radius. |

This is a minimum model. Add malicious insiders, compromised browsers/apps, XSS, SSRF, supply-chain compromise, signing-key theft, tenant administrators, and operational access when the deployment makes them realistic. Record which controls collapse if an attacker gets both a token and its proof key.

## Attack-to-control matrix

| RFC section | Attack and precondition | Security result | Decisive mitigations | Minimum negative test |
|---|---|---|---|---|
| [§4.1](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.1) | **Insufficient redirect validation.** Wildcards, patterns, suffix matching, URI normalization, dangling subdomains, or a callback open redirect let an attacker select a destination. | Code or token reaches an attacker; client identity/authentication is effectively bypassed. | Full pre-registration and exact string comparison; native loopback port is the only exception; no open redirectors; prefer code flow. | Register one exact callback, then try sibling subdomain, suffix-confusion host, changed scheme/port/path/query, encoded forms, userinfo, fragment, and nested redirect. All but the precise allowed loopback-port variation fail before user interaction. |
| [§4.2](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.2) | **Referer leakage.** Authorization or callback page loads or links to attacker-controlled content while the URL contains request/response artifacts. | Attacker learns code, state, and in buggy cases a fragment token; code injection/CSRF/replay follows. | No third-party content or external links; restrictive Referrer Policy; code flow; PKCE; one-use code/state; form-post response mode where appropriate. | Load a controlled third-party image/link from auth and callback pages and assert the request contains no sensitive query/fragment-derived data or token. |
| [§4.3](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.3) | **Browser-history leakage.** Codes or tokens remain in navigated URLs; query-token transport makes tokens durable in history/logs. | Local/device attacker or later browser user replays the artifact. | Never query-parameter access tokens; code single-use + PKCE; form-post where selected; callback URL cleanup is defense in depth, not the primary control. | Complete the flow, inspect history and navigation entries, and verify no access token appears; replay a captured code and require failure. |
| [§4.4](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.4) | **Mix-up.** Client supports multiple issuers; attacker controls one and causes an honest issuer's response to be sent to attacker endpoints selected from the wrong issuer context. | Attacker receives an honest code/token or client credentials; malicious endpoint attack. | Bind chosen issuer to session; validate response `iss` under RFC 9207 or signed equivalent; distinct per-issuer callback only as fallback; bind all discovered endpoints and keys to one issuer. | Start with issuer A, return a response labeled B or swap B's token endpoint/JWKS into A's metadata. Client aborts before code exchange or key use. |
| [§4.5](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.5) | **Authorization-code injection.** Attacker steals a code and injects it into the attacker's own client session; legitimate client performs authenticated exchange. | Attacker session becomes bound to victim resources/identity. | PKCE for public and preferably confidential clients; confidential OIDC `nonce` only with token-endpoint validation precautions; exact redirect/code binding; prevent response leakage. | Start two sessions and swap their authorization codes. The callback fails and no application session or token use occurs. Repeat across devices/browser profiles. |
| [§4.6](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.6) | **Access-token injection.** In a token-at-authorization-endpoint flow, attacker injects a stolen token plus known transaction state. | Victim's client operates with attacker-selected/victim token; identity or resource confusion. | Avoid implicit/token responses. Pure OAuth has no general transaction binding for the injected access token; OIDC `at_hash` + signed ID Token/`nonce` can help, but sender constraint is still needed for direct replay. | If such a flow remains, substitute a valid token from another transaction and require rejection before API use. Prefer proving the flow is disabled. |
| [§4.7](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.7) | **Callback CSRF/login CSRF.** Attacker sends an authorization response/code from the attacker's transaction into a victim's client session. | Victim is logged into or connects the attacker's account; attacker can later observe victim actions/data. | Verified PKCE binding, or OIDC nonce, or one-time session-bound state; protect application data carried in state. | Missing, wrong, replayed, expired, swapped, and cross-browser state/nonce/verifier values all fail closed before session creation. |
| [§4.8](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.8) | **PKCE downgrade.** Attacker removes `code_challenge`; client later sends a verifier; server treats the code as non-PKCE and accepts it. | CSRF/code substitution despite a client believing PKCE is active. | Server records challenge presence and rejects verifier-without-original-challenge; mandate PKCE per client where possible. | Obtain a code from a request without a challenge and redeem it with any `code_verifier`. Require rejection. Also reject missing/wrong verifier when a challenge was present. |
| [§4.9](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.9) | **Counterfeit or compromised resource server.** Dynamic client sends a broadly valid token to an attacker resource, or a compromised resource obtains reusable tokens. | Token is replayed at another resource server or mined from logs/storage. | Audience/resource restriction, sender constraint, resource URL binding, token-secret handling, no plaintext logs/storage. | Request a token for resource A and present it to B; B rejects. Send a B-intended token to a controlled counterfeit A and confirm it cannot be replayed at B. |
| [§4.10](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.10) | **Stolen-token misuse.** A5 holds an otherwise valid bearer token. | Direct API access until expiry/revocation; cross-resource blast radius. | DPoP or mTLS sender constraint plus proof replay controls; audience restriction and least privilege. | Present token without proof, with wrong key/certificate, wrong `htu`/`htm`, stale/replayed proof, and at wrong audience. Every case fails at the resource server. |
| [§4.11](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.11) | **Open redirection/phishing.** Client or authorization server forwards arbitrary destinations, including attacker-registered redirect URIs after error/denial/silent requests. | Artifact exfiltration and trusted-domain phishing. | No client open redirect; AS does not redirect invalid client/URI combinations; authenticate before redirect; risk-assess attacker-provided redirects and do not auto-follow untrusted ones. | Feed callbacks, login return URLs, error paths, logout returns, and consent-denial paths external or encoded URLs. Require allowlisted destinations or a non-redirecting error/interstitial. |
| [§4.12](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.12) | **307 credential forwarding.** AS returns 307 after a credential-bearing POST, preserving method/body to the client callback. | Client receives the user's AS credentials. | Never use 307; use 303 when HTTP redirection follows credential submission. | Submit synthetic credentials and assert callback receives GET without the form body. Fail any 307 response on the path. |
| [§4.13](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.13) | **Proxy-header spoofing/internal interception.** Outside caller supplies trusted certificate, scheme, host, client IP, or token-binding headers; internal hop is unprotected. | Bypass client authentication, redirect checks, security policy, or proof binding. | Proxy strips/overwrites external security headers; authenticates upstream/downstream; encrypts and replay-protects internal hop. | Send each trusted header from the public side and verify the application sees only proxy-generated values. Test direct-app access is blocked and internal TLS identity is checked. |
| [§4.14](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.14) | **Refresh-token theft/replay.** Long-lived refresh token mints new access tokens; naive rotation accepts an older token independently. | Persistent access and privilege expansion. | Risk-based issuance; consented scope/resource binding; sender constraint or rotation with family reuse detection for public clients; inactivity expiry; integrity of self-contained tokens. | Use token R1 to obtain R2, then replay R1. The family/grant is revoked so R2 cannot continue. Test scope/resource expansion and wrong-client/sender use. |
| [§4.15](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.15) | **Client impersonates user.** Client-chosen ID collides with a user `sub`, and a resource server conflates client-credentials and user grants. | Malicious client accesses user-owned resources. | Separate namespaces/token types/claims or unambiguously signal grant subject type; prevent client-chosen collisions. | Issue user and client-only tokens with colliding candidate identifiers. Resource authorization must distinguish them and deny client-only access to user resources. |
| [§4.16](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.16) | **Clickjacking.** Authorization/login/consent UI is framed or overlaid by an attacker. | User leaks credentials or approves attacker-selected access. | CSP `frame-ancestors` on every relevant page; compatible `X-Frame-Options`/additional defenses; intentional per-client framing policy only where required. | Attempt framing auth, login, consent, device, and error pages from untrusted and explicitly trusted origins. Verify policy on success and error responses. |
| [§4.17](https://www.rfc-editor.org/rfc/rfc9700.html#section-4.17) | **In-browser message leak/injection.** Sender uses `*` or attacker-selected origin; receiver does not validate exact sender origin. | Code/token leaked to malicious window or forged response accepted. | Exact pre-registered target origin; exact sender-origin comparison; no wildcard; ordinary CSRF/issuer/transaction controls still apply. | Send from sibling, prefix/suffix, alternate port/scheme, `null`, and untrusted origins; send to wildcard; all are rejected and no artifact is disclosed. |

## Adversarial test suites

### Authorization server / OpenID Provider

- Redirect registration and authorization requests: exact match, invalid encoding, duplicate parameters, ambiguous parser inputs, loopback exception limits, and error paths.
- PKCE matrix: missing, `plain`, `S256`, wrong verifier, missing verifier, verifier without original challenge, reused challenge, and challenge/client mix-up.
- Code lifecycle: first redemption, concurrent double redemption, expired code, wrong client, wrong redirect URI, wrong issuer, and descendant revocation behavior.
- Refresh lifecycle: wrong client/sender, scope or resource escalation, replay of prior generation, concurrent refresh, family revocation, inactivity expiry, logout/password-change policy.
- UI responses: 303 vs 307, CSP and framing, third-party subresources, Referrer Policy, cache headers, errors, denial, and silent authentication.
- Metadata: exact issuer, correct endpoints/JWKS, supported PKCE methods, stable HTTPS URLs, key rollover, stale cache, and no attacker-controlled endpoint substitution.

### Client / relying party

- Start two or more concurrent transactions; swap `state`, nonce, verifier, issuer, code, and browser session independently.
- Reject callbacks that are unsolicited, duplicated, expired, on the wrong route, or delivered with both success and error parameters.
- Prove no token is used and no application session exists before every required issuer/ID Token/nonce/hash check completes.
- Test multi-issuer endpoint and key separation, including a malicious discovery document that reuses an honest authorization endpoint.
- Verify token values never enter URLs, page markup, Referer data, browser/server logs, error trackers, analytics, or exception messages.

### Resource server

- Reject wrong issuer, token type, audience/resource, scope/action, expiry/not-before, sender key/certificate, malformed proof, replayed proof, and client-vs-user subject confusion.
- Exercise opaque-token introspection failures and cache behavior where used; do not treat inactive/unknown as success.
- Verify every endpoint enforces authorization independently, including aliases, batch operations, WebSockets, downloads, and background jobs.
- Confirm raw authorization headers and token bodies are redacted before logs, traces, metrics, error reports, and support tooling.

## Safe testing rules

- Test only systems and tenants within the user's authorization. Never probe a third-party identity provider or production account merely because an attack is described here.
- Prefer local, disposable, or dedicated test deployments and synthetic users/resources.
- Use clearly fake token markers where cryptographic validity is irrelevant. When real tokens are required, minimize privilege/lifetime and redact them from all output.
- Do not paste tokens, secrets, authorization codes, cookies, private keys, or customer identity data into prompts, issue trackers, fixtures, screenshots, or verifier briefs.
- Avoid destructive revocation/rotation in production unless explicitly authorized and coordinated. During incidents, preserve evidence and follow the organization's response process.
- A negative test passes only when the system fails closed before the protected action. A later error after a session or API call was already authorized is not a pass.
