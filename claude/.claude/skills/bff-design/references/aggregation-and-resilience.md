# Aggregation and Resilience

How a BFF fans out to upstream services and stays standing when they fail. The BFF is the only tier that knows which response sections are optional — semantic resilience cannot be fully delegated to a mesh or SDK.

## Composition

- **Fan out in parallel.** Sequential upstream calls stack latencies; run every independent call concurrently and compose (Newman). Model the dependency graph explicitly — B-needs-A chains are sequential by nature, everything else is not.
- **Batch and memoize per request.** Cross-service N+1 (a list of orders, then one customer lookup per order) is solved with a request-scoped batcher (the DataLoader pattern): coalesce loads within one execution frame into one batch call. The batch cache is strictly per-request — a process-lifetime instance leaks one user's data into another's response.
- **Know the in-memory-join limit.** API composition joins small result sets in BFF memory. Paginating or sorting a merged union of two large upstream lists is the documented failure mode (Richardson: "inefficient, in-memory joins of large datasets") — that query wants a pre-joined read model (CQRS; see `event-sourcing` for the full pattern), not a cleverer BFF.
- **Aggregation with domain logic is a service, not a BFF feature.** Azure's line: custom domain logic, complex transformations, or long-running orchestration belongs in a dedicated service behind the gateway. The BFF composes and shapes; when composition grows rules, push it down (the rule-of-three ladder in SKILL.md).

## Partial Failure

Decide **per endpoint, in the contract**, whether each section is required or optional — and make degraded responses a predictable shape, not an accident (Azure: "make this decision explicit in the policy design so that clients experience predictable behavior").

- Required section fails → distinguish *why*. A dependency failure or deadline exhaustion fails the whole request with one problem-details response (502/504) and no partial payload — predictability over cleverness. An *expected* upstream outcome — not-found, forbidden, conflict, validation — is not a dependency failure: translate it into the BFF's own contract statuses as normal. A required section is fresh-or-fail by default; serving it stale is allowed only when the contract explicitly marks it stale-tolerable with a maximum age (account balances: no; a product catalog header: perhaps).
- Optional section fails → degrade through the fallback chain **as declared per section in the contract**: each section states which rungs it permits — stale cache (with a maximum stale age; staleness-unsafe data like entitlements, limits, or market state prohibits this rung outright) → static/safe default → empty-with-error. Whenever stale data is emitted, `asOf` is mandatory, not decorative. (Netflix's taxonomy maps onto the chain, not beside it: fallback-via-cache = rung 1, static/stubbed fallback = rung 2, fail-silent = rung 3; fail-fast is what the fast-fail stage already did.) *No fallbacks for writes*; a failed write must propagate.
- Represent partiality — including staleness — explicitly. GraphQL codifies partial data (`data` + `errors`); the REST-BFF analog is per-section state beside the sections that succeeded: a failed section carries an RFC 9457 problem-details object, and a degraded section carries a marker (`degraded: { source: 'stale-cache' | 'default', asOf: ... }` — `asOf` required whenever the source is stale) so a dashboard never silently presents stale numbers as fresh. This is a composition of standards, not itself standardized — say so in the contract docs.
- Per-section recovery: give the client a per-section refetch (query parameter or section endpoint); a refetch is a new request under the same retry-ownership rules, not an excuse for a client retry loop.
- The frontend must be built to render partial payloads; a BFF that degrades gracefully in front of a client that blank-screens on a missing key has moved the outage, not prevented it. Test the degraded renders (see `front-end-testing`).

## Resilience Budgets

The numbered stages are the *construction* order, not the runtime order. At runtime there are two nested pipelines, and their composition is where overload protection is won or lost: the **outer, per-logical-request pipeline** runs admission control/shedding *first* (reject sheddable work before any of it is admitted — Google SRE), then sets the deadline; the **inner, per-attempt pipeline** runs fast-fail check → bulkhead slot → timeout for each attempt — breaker *before* bulkhead, or new work queues behind slow attempts only to discover the circuit is open (Azure's breaker guidance: open-circuit calls fail immediately). Accounting rules that make the nesting safe: an open-circuit rejection is terminal to the retry loop, never retried; a bulkhead rejection is a *local* capacity signal — it consumes no retry budget and is never counted as a dependency failure by the breaker, which samples only actual dependency attempts; half-open probe admission is deliberate (one probe, not a thundering herd); a backoff wait holds no bulkhead slot (release between attempts); a hedge consumes both a retry-budget token and its own bulkhead slot — it is extra concurrency and must be paid for as such.

First, enumerate the edges — the ownership rules below are per edge, and forgetting an edge is how three retry sites accumulate:

| Edge | Retry owner | Form |
|------|-------------|------|
| SPA → BFF | The SPA's fetch wrapper | At most one jittered retry, connection-level failures only — never on a 5xx that already carries a degraded payload. (Traffic-ratio budgets are server-side aggregates; a browser tab uses max-attempts + jitter.) |
| BFF → upstream (semantic) | The BFF | Budgeted retries as below; a hedge counts as a retry against the same owner's budget |
| Mesh under either edge | Nobody, unless delegated | If the BFF owns an edge's retries, mesh retries on that edge are off. Delegating connection-level retries to the mesh is allowed only *instead of*, never in addition to, BFF retries on that edge — one owner means one, not one per failure class |

The spine, each stage feeding the next:

1. **Deadline per request, propagated.** Derive the end-to-end deadline from the page's latency SLO/UX budget (it is a property of the endpoint, configured at the entry point — not something the SPA sends per request); every upstream call carries the remaining budget, reduced by time already spent (Google SRE). Effective per-call timeout = min(the empirical timeout below, remaining deadline); a retry attempt draws from the remaining budget, never a fresh one. Without propagation, upstreams burn capacity on requests the user already abandoned. Check cancellation before expensive work.
2. **Empirical per-upstream timeouts.** Choose an acceptable false-timeout rate and read the timeout off the latency distribution — e.g. 0.1% budget → p99.9 of that upstream's observed latency (AWS/Brooker). Never copy-paste a global 30s.
3. **One retry owner per edge** (table above). Layered retries multiply: three retries at each of three layers is up to 4³ attempts per user action — a self-inflicted denial of service (Google SRE; Linkerd). Budgeted, not counted, where the owner is a server: cap retries as a ratio of live traffic (~10% Google, 20% Linkerd default), with full-jitter exponential backoff (Brooker: full jitter beats equal jitter decisively). What to retry is a decision matrix, not a status-class slogan: retry transport/connect failures and retryable 5xx; honor `Retry-After` on 429/503 against the remaining deadline; treat other 4xx as contract errors that retrying cannot fix (408/425 have their own RFC semantics — decide them per operation). Retry side-effecting calls only with idempotency keys and a tested deduplication contract (`api-design`); cap both attempts and total elapsed time by the propagated deadline.
4. **Bulkhead per upstream.** Bound each upstream's concurrency separately so one latent dependency cannot exhaust the BFF's event loop, sockets, or memory (Hystrix's founding argument). Estimate the starting bound from *mean* offered load × *mean* service time (Little's law relates means — sizing from peak×p99 inflates the limit exactly when latency explodes), then set the real limit and queue bound from downstream capacity, memory/socket budgets, and your rejection SLO, confirmed under load test. One slow service must not queue the whole page.
5. **Fast-fail the known-bad.** A circuit breaker (or mesh outlier detection — pick **one fast-fail owner per upstream**, not both) converts a down upstream into an instant local failure, which is what triggers the partial-failure fallback chain instead of a timeout-long hang. This is a different tool from stage 3's retry token bucket (which bounds retry *volume* but never rejects healthy first attempts) and from overload admission control (stage 7). Specify the breaker concretely: failure classes counted, sampling window and minimum volume, open duration, half-open probe policy, instance scope (per upstream, shared across requests — a per-request breaker does nothing). A fast-fail rejection consumes no retry budget; it is the budget working.
6. **Hedge only the idempotent tail.** Send a second request after the p95 latency has elapsed; cancel the loser — ~5% extra load for a collapsed tail (Dean & Barroso). Reads only.
7. **Shed by criticality.** Under overload, reject sheddable work early and serve degraded responses; propagate request criticality downstream so upstreams shed the same work. Criticality is *assigned by the BFF* from server-owned endpoint policy — strip any externally supplied criticality/baggage at the boundary; a browser-settable priority header is a caller-controlled fast lane (Google's model assigns criticality at the frontend tier, never trusts the client). Carry it on an allowlisted internal header (W3C baggage as the transport is fine once inbound values are stripped). Degraded paths rot — exercise them under synthetic overload regularly (Google SRE).

**Mesh coexistence** goes beyond retries: mesh timeouts on a BFF edge must exceed the BFF's own per-call timeout (the BFF is the decider, the mesh the backstop), and mesh outlier detection overlaps breaker duty — assign fast-fail per edge just like retries. A mesh whose config fights the BFF's budgets will re-create every symptom this spine removes.

**Single-upstream BFFs** (one general-purpose API behind the BFF — the common starting point): the spine collapses rather than disappears — one deadline, one empirical timeout, one retry owner, one fast-fail. What never collapses is overload protection: a single slow dependency still exhausts workers, sockets, and memory, so the BFF keeps a global concurrency bound, bounded queues, and load shedding even with one upstream; only the *per-upstream partitioning* waits for a plural backend.

## Caching

Identity in the key, or not cached at all:

- **Tier 1 — request-scoped memoization** (DataLoader-style): always safe, dies with the request.
- **Tier 2 — in-BFF shared cache**: key on `{tenant, user-or-attribute, resource}`, with TTL set by data volatility and product tolerance (seconds-scale for money, minutes-scale for catalogs — a number someone chose, not a default someone inherited). Cache on the *coarsest attribute actually used*: a response depending only on the user's role can be shared across users with that role; a response using the user ID cannot (Fastly/FT practice). Getting identity out of the key is the cache-poisoning/leak vector (PortSwigger) — and CDN `Vary` handling is configuration-dependent (verify yours; `api-design` covers standard `Vary` semantics), so for identity-varying responses make the key explicit rather than trusting header inference, or keep them out of shared caches entirely.
- **Tier 3 — HTTP/CDN caching**: only for responses provably not per-user, or keyed per RFC 9111 (shared caches must not reuse `Authorization`-bearing responses unless the response explicitly permits it via `public`, `s-maxage`, or `must-revalidate`; `private` marks single-user responses). Keep session-bearing responses `no-store` — `bff-entry-points` mandates this on the protected surface; verify it rather than assuming it.
- **Staleness as resilience**: `stale-while-revalidate` hides refresh latency; `stale-if-error` (RFC 5861) is the HTTP-native "serve the cached section when the upstream is down" — the cheapest fallback in the partial-failure chain.
- **Aggregate freshness = freshest ingredient.** An aggregated response's cache lifetime is bounded by its shortest-lived section (Newman); don't let a day-cacheable catalog section drag a live inventory count along with it.

## Streaming Composition

When the BFF composes upstream event streams into client-facing SSE/WebSocket (protection and registration live in `bff-entry-points`; endpoint-vs-workflow placement in `structure-codebase`):

- Own an ordered event log per stream: SSE resume (`Last-Event-ID`) is a single opaque cursor, so composing multiple upstream streams means mapping their cursors into one monotonic client-facing sequence — assign an `id:` to every event or replay silently breaks.
- On reconnect beyond the replay window, send snapshot-then-deltas rather than unbounded replay (established event-stream practice; see `event-sourcing` projections).
- Neither SSE nor WebSocket gives *application-level* end-to-end backpressure — TCP flow control slows the socket, but nothing tells the producer a consumer's processing is behind. Bound per-client buffers, coalesce/conflate superseded events (keep latest state, drop intermediates), and disconnect-with-resync when a consumer can't keep up.
